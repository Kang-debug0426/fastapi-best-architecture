## 一、FastAPI

### ✅ Day1 已理解

**FastAPI 是什么**：和 Flask 一样是 Web 框架，负责"接收请求、执行逻辑、返回结果"。区别是 FastAPI 更现代：自动生成 Swagger 文档、原生支持异步、性能更高。

**应用入口**：`backend/main.py` → `app = register_app()` → `backend/core/registrar.py` 第83行 `app = FastAPI(...)` 才是真正创建实例的地方。

**为什么 prod 环境看不到 Swagger**：`registrar.py` 里 `openapi_url=settings.FASTAPI_OPENAPI_URL`，prod 环境下这个值被设为 `None`，FastAPI 不生成文档路由。

**JWT 登录流程**：
```
POST /api/v1/auth/login/swagger (username + password)
  → FastAPI 接收
  → PostgreSQL 查 sys_user 表，比对密码哈希
  → 生成 JWT Token（三段：Header.Payload.Signature）
  → Redis 存入 session_uuid（用于踢下线）
  → 返回 access_token + 用户信息
```

**JWT Token 结构**：
- 第一段 Header：加密算法（HS256）
- 第二段 Payload：用户ID(sub)、过期时间(exp)、session_uuid
- 第三段 Signature：用服务器密钥签名，防篡改
- Payload 可以被任何人解码，但无法伪造签名

**密码为什么不返回给浏览器**：没有任何理由需要把密码发给前端，安全设计原则——最小必要信息。

### 1.1 路由（Router）

#### 核心概念

- `@app.get/post/put/delete("/path")` 对应 HTTP 四种方法
- **路径参数**：`/hello/{name}` → 函数参数 `name: str`，值在 URL 路径里
- **查询参数**：`/search?keyword=x&limit=10` → 函数参数 `keyword: str, limit: int`，值在 `?` 后面
- **请求体**：`POST /users` → 函数参数 `data: dict` 或 `user: UserCreate`，值在 HTTP body 里
- **类型注解决定 FastAPI 从哪里取值**：`str/int` → 路径或查询；`BaseModel` → 请求体
- `APIRouter(prefix="/users", tags=["users"])`：prefix 管路径前缀，tags 管 Swagger 分组
- `app.include_router(router)` 把子路由挂到主 app
- `routers/` 文件夹下必须有 `__init__.py`（哪怕是空的），Python 才认这是包

#### 踩坑记录

**坑1**：类型注解写成 `name:name` 而不是 `name:str`
- 原因：误以为冒号后面写变量名，实际上冒号后面写**类型**（str/int/float/bool）
- 解决：`def accept(name: str):`

**坑2**：uvicorn 在错误目录启动
- 报错：`Error loading ASGI app. Could not import module "hello"`
- 原因：在主项目根目录跑 `uvicorn hello:app`，但 hello.py 在 sandbox/day2/ 里
- 解决：先 `cd sandbox/day2` 再启动

**坑3**：`reload=Ture` 拼写错误
- 报错：`SyntaxError: expected argument value expression`
- 解决：改成 `reload=True`

**坑4**：`routers/` 缺少 `__init__.py`
- 报错：`ModuleNotFoundError` 或 `Attribute "app" not found`
- 解决：在 `routers/` 下新建空文件 `__init__.py`

#### 代码片段

```python
# APIRouter 标准写法
from fastapi import APIRouter
router = APIRouter(prefix="/users", tags=["users"])

@router.get("/list")
def get_user_list():
    return {"users": ["luchia", "alice"]}

# main.py 挂载
from fastapi import FastAPI
from routers import users, items
app = FastAPI()
app.include_router(users.router)
app.include_router(items.router)
```

### 1.2 依赖注入（Depends）

#### 核心概念

- `Depends(fn)` 传函数本身，**不加括号**（加括号是调用，不加是传引用）
- FastAPI 看到 `Depends(fn)` 自动调用 fn，把返回值注入给参数
- **yield 依赖**：yield 前=请求前执行（开资源），yield 值=注入值，yield 后=请求结束执行（关资源）
- **Header(...)**：从 HTTP 请求头取值，`...` 表示必填，不传返回 422
- **依赖嵌套**：依赖函数自己也可以有 Depends，FastAPI 自动从最底层开始解析
- 企业项目里 `CurrentSession` = `Annotated[AsyncSession, Depends(get_db)]` 的类型别名

#### 踩坑记录

**坑1**：忘记写 `app = FastAPI()`
- 报错：`NameError: name 'app' is not defined`
- 原因：导入了 FastAPI 但没有创建实例，直接用 `@app.get` 装饰器
- 解决：在所有路由之前加 `app = FastAPI()`

**坑2**：yield 执行顺序理解错误
- 误解：以为 yield 之后的代码要"等下次请求"才执行
- 正确：yield 之后的代码在**同一次请求结束后**自动执行，不需要任何额外操作
- 顺序：请求进来 → yield 前代码 → yield 值注入接口 → 接口执行完返回响应 → yield 后代码

**坑3**：`@field_validator` 写在类外面
- 报错：校验器不生效，传含空格的 username 也能通过
- 原因：装饰器和方法没有缩进到类里面，变成了模块级函数
- 解决：`@field_validator` 和 `def` 都要缩进一级，在类的内部

#### 代码片段

```python
from fastapi import FastAPI, Depends, Header

app = FastAPI()

# yield 依赖（管资源生命周期）
def get_db():
    print("数据库连接已打开")
    yield {"db": "fake_session"}
    print("数据库连接已关闭")  # 请求结束后自动执行

# Header 依赖（从请求头取 token）
def verify_token(token: str = Header(...)):
    return token

# 依赖嵌套
def get_current_user(token = Depends(verify_token)):
    return {"username": "luchia", "token": token}

@app.get("/me")
def read_me(user = Depends(get_current_user)):
    return user
```

## 一点五、Pydantic（Day 3）

### 核心概念

- `BaseModel` 是所有 Schema 的基类，继承它就获得自动校验能力
- 字段定义：`字段名: 类型 = Field(默认值或..., 校验参数)`
- `...` 表示必填（不能不传）
- **请求 Schema vs 响应 Schema 要分开**：响应不能把密码等敏感字段返回给前端
- `response_model=UserResponse` 自动过滤掉响应里多余的字段
- `EmailStr` 需要 `pip install pydantic[email]`

### Field 常用参数

| 参数 | 适用类型 | 含义 |
|------|---------|------|
| `min_length` / `max_length` | str | 字符串长度限制 |
| `ge` / `le` | int/float | 大于等于 / 小于等于 |
| `gt` / `lt` | int/float | 大于 / 小于 |
| `description` | 所有 | Swagger 文档里显示的字段说明 |

### 踩坑记录

**坑1**：`@field_validator` 写在类外面不生效
- 原因：没有缩进，变成模块级函数，Pydantic 不知道它属于哪个类
- 解决：缩进一级放到类里面

**坑2**：校验器里 `if '' in v` 判断空字符串而不是空格
- 原因：`''` 是空字符串，`' '` 才是空格
- 解决：`if ' ' in v:`

**坑3**：校验器忘记 `return v`
- 后果：字段值变成 None
- 规则：校验通过必须 return v，校验失败 raise ValueError

### 代码片段

```python
from pydantic import BaseModel, Field, field_validator, EmailStr

class UserCreate(BaseModel):
    username: str = Field(..., min_length=3, max_length=20, description="用户名")
    age: int = Field(..., ge=0, le=150, description="年龄")
    email: EmailStr

    @field_validator("username")
    @classmethod
    def username_no_spaces(cls, v):
        if " " in v:
            raise ValueError("username 不能包含空格")
        return v  # 必须 return！

class UserResponse(BaseModel):
    id: int
    username: str
    # 没有 age、email —— 不返回给前端

# 接口里用法
@app.post("/users", response_model=UserResponse)
def create_user(user: UserCreate):
    return {"id": 1, "username": user.username}
```

### 1.6 JWT 认证（阶段4）

#### 核心概念

**JWT 是什么**：JSON Web Token，登录成功后服务器生成的一串加密字符串，交给客户端保存。

**三段结构**：`Header.Payload.Signature`
- Header：算法类型（HS256）
- Payload：用户信息（sub=username, exp=过期时间）
- Signature：用密钥签名，防篡改

**关键理解**：JWT 是编码不是加密，任何人都能解码 Payload，但没有密钥无法伪造 Signature。

**生成 token**：
```python
from jose import jwt
from datetime import datetime, timedelta

SECRET_KEY = "your-secret-key"
ALGORITHM = "HS256"

payload = {
    "sub": username,           # 用户标识
    "exp": datetime.utcnow() + timedelta(hours=1)  # 过期时间
}
token = jwt.encode(payload, SECRET_KEY, algorithm=ALGORITHM)
```

**验证 token**：
```python
payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
username = payload.get("sub")
```

**只用 JWT 的问题**：token 存在客户端，服务器无法主动撤回。用户被封号、强制下线时，只能等 token 自然过期。解决方案：加 Redis Session（阶段4 Task 4-2）。

**应用场景**：用户登录后，每次请求带上 token，服务器验证 token 确认身份，不需要每次查数据库。

#### 踩坑记录

**坑1**：路由路径不加 `/` 开头
- 错误：`@app.post("auth/login")`
- 后果：Swagger 拼 URL 时变成 `http://127.0.0.1:8000auth/login`，缺少斜杠
- 解决：永远以 `/` 开头：`@app.post("/auth/login")`

**坑2**：Swagger UI 的 Header 参数输入框不生效
- 现象：在 Swagger 里填了 authorization，curl 里看不到这个 header
- 原因：Swagger 对自定义 Header 支持不好
- 解决：用 curl 直接测试 Header 认证

**坑3**：Swagger 里 authorization 输入框填了 `authorization: Bearer xxx`- 现象：一直 422 或请求失败
- 原因：输入框只填 Header 的值，不要填 Header 的名字
- 解决：只填 `Bearer eyJhbG...`，不要带 `authorization: ` 前缀

**坑4**：Redis key 格式不一致导致 session 验证永远失败
- 现象：登录成功，访问受保护接口一直 401
- 原因：login 存 key 时用 `ainotes:token:{session_id}`，verify_token 查 key 时用 `ainotes:token:{username}:{session_id}`，格式不一样找不到
- 解决：统一 key 格式，存和查都用 `ainotes:token:{username}:{session_id}`

**坑5**：verify_token 里误写成 `r.delete` 而不是 `r.exists`
- 现象：第一次访问成功，第二次就 401
- 原因：verify_token 是验证函数，不应该删 key；每次验证都把 session 删掉了
- 解决：改成 `if not r.exists(key): raise HTTPException(401)`

**坑6**：logout 函数里忘记取 username，key 拼错删不掉
- 现象：logout 返回成功，但同一 token 还能继续访问
- 原因：`r.delete(f"ainotes:token:{session_id}")` 少了 username，key 不存在，什么都没删
- 解决：先 `username = payload.get("sub")`，再拼完整 key

### 1.7 密码哈希（bcrypt）

#### 核心概念

**为什么不能明文存密码**：数据库泄露时，明文密码直接暴露。哈希后即使泄露，也无法反推原始密码。

**bcrypt 的特点**：
- 单向哈希，不可逆（不能从哈希值还原密码）
- 每次哈希结果不同（因为有随机 salt）
- 验证时用 `checkpw` 对比，不是直接比较字符串

**用法**：
```python
import bcrypt

# 加密（注册时）
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt()).decode()

# 验证（登录时）
is_valid = bcrypt.checkpw(password.encode(), hashed.encode())
```

**应用场景**：用户注册时哈希密码存数据库，登录时用 `checkpw` 验证，永远不存明文密码。

#### 踩坑记录

**坑1**：passlib 和新版 bcrypt 不兼容
- 报错：`AttributeError: module 'bcrypt' has no attribute '__about__'`
- 原因：passlib 读取 bcrypt 版本的方式在新版本失效
- 解决：卸载 passlib，直接用 `bcrypt` 库，`pip install bcrypt==4.0.1`

### 1.8 请求头（Header）与依赖注入配合

#### 核心概念

**Header 取值**：
```python
from fastapi import Header

def verify_token(authorization: str = Header(...)):
    # authorization 的值是请求头里 Authorization 字段的内容
    # 通常格式是 "Bearer eyJhbG..."
    token = authorization.replace("Bearer ", "").strip()
```

**`Header(...)` 的含义**：`...` 表示必填，请求头里没有这个字段直接返回 422。

**和 Depends 配合保护接口**：
```python
def verify_token(authorization: str = Header(...)) -> str:
    # 验证逻辑...
    return username  # 必须 return，否则注入值是 None

@app.get("/notes")
async def get_notes(username: str = Depends(verify_token)):
    # username 是 verify_token 的返回值
    return list(notes.values())
```

**应用场景**：任何需要登录才能访问的接口，加 `Depends(verify_token)` 即可保护，不需要在每个接口里重复写验证逻辑。

#### 踩坑记录

**坑1**：`verify_token` 函数忘记 `return username`
- 后果：`Depends(verify_token)` 注入的值是 None，接口拿不到用户信息
- 规则：依赖函数必须 return 值，FastAPI 才能把它注入给接口参数

**坑2**：`authorization.split("Bearer ")[1]` 在没有前缀时报 IndexError
- 原因：如果直接传 token 字符串（没有 "Bearer " 前缀），split 结果只有一个元素
- 解决：用 `authorization.replace("Bearer ", "").strip()` 更安全

## 二、Docker

### 2.1 镜像（Image）

#### 核心概念

**镜像 = 一个打包好的"软件 + 运行环境"模板**。
- 类比：镜像是"光盘"，容器是"装在电脑里运行起来的程序"
- 镜像是只读的，启动后变成容器才可以读写
- 镜像由多层（layers）叠加组成（Union FS），每层只记录"和上一层的差异"，所以多个镜像可以共享底层

**Day 1 实际见到的镜像**：
- `python:3.10-slim-bookworm` — Python 运行环境基础镜像
- `postgres:16` — 官方 PostgreSQL 数据库镜像
- `redis:latest` — 官方 Redis 镜像
- `nginx:latest` — 官方 Nginx 镜像
- `rabbitmq:3.13.7` — 官方 RabbitMQ 镜像
- `fba_server:latest` — 本地用 Dockerfile 构建出来的项目镜像
- `ghcr.io/astral-sh/uv:python3.10-bookworm-slim` — Dockerfile 第一阶段用的构建镜像

**镜像怎么来的**：
1. **拉取**：`docker pull postgres:16`（从 Docker Hub 下载）
2. **构建**：`docker build -f Dockerfile .`（根据 Dockerfile 自己造一个）
3. **Compose 自动构建**：`docker compose up` 时，有 `build:` 配置的服务会自动构建

#### 踩坑记录

**坑：在国内拉镜像超时/EOF**
- 原因：Docker Hub 在国内访问不稳定，被墙
- 解决：科学上网 + Clash Verge 开 TUN 模式（让 Docker 流量也走代理）
- 错误方向：不要再改 `registry-mirrors` 国内镜像源（基本都失效）

**坑：Dockerfile 里 `ADD https://...` 失败**
- 原因：构建过程中要联网下载文件
- 解决：改用多阶段构建，从前一阶段 `COPY --from=builder` 复制二进制
- 实例：本项目把 `ADD https://astral.sh/uv/install.sh` 改为 `COPY --from=builder /usr/local/bin/uv`

#### 常用命令

| 命令 | 作用 | 是否危险 |
|------|------|---------|
| `docker images` | 列出本地所有镜像 | 只读，安全 |
| `docker pull <镜像名>` | 从仓库拉取镜像 | 只下载，安全 |
| `docker rmi <镜像ID>` | 删除镜像 | **会删除，谨慎** |
| `docker build -t <名称> .` | 根据 Dockerfile 构建镜像 | 安全（生成新镜像） |
| `docker history <镜像>` | 看镜像的层级历史 | 只读，安全 |

### 2.2 容器（Container）

#### 核心概念

**容器 = 镜像的"运行实例"**。
- 类比：镜像是 Word 模板，容器是基于这个模板新建的具体文档
- 一个镜像可以同时跑多个容器（同一份模板可以新建很多文档）
- 容器是隔离的：每个容器有自己的文件系统、网络、进程空间，互相不影响
- **容器不是虚拟机**：容器和宿主机共享同一个内核（Linux 内核），所以启动快、资源开销小

**容器的生命周期**：
```
镜像 → docker run → 容器创建并运行 → docker stop → 容器停止（不删除）
                                  → docker start → 重新运行
                                  → docker rm → 容器删除
```

**Day 1 跑起来的容器**（`docker compose ps` 可看）：
| 容器名 | 角色 | 暴露端口 |
|--------|------|---------|
| fba_server | FastAPI 后端 | 8001 |
| fba_postgres | 数据库 | 5432 |
| fba_redis | 缓存/Session | 6379 |
| fba_rabbitmq | 消息队列 | 5672 / 15672 |
| fba_nginx | 反向代理 | 8000 |
| fba_celery_worker | 异步任务执行器 | - |
| fba_celery_beat | 定时任务调度 | - |
| fba_celery_flower | Celery 监控面板 | 8555 |
| fba_grafana | 监控面板 | 3000 |
| 等等... | | |

**关键理解**：容器之间通信用"容器名"作为 hostname，比如 fba_server 连数据库写的是 `fba_postgres:5432`，不是 `localhost:5432`。

#### 踩坑记录

**坑：`docker compose restart` 不会重新加载挂载的 .env 文件**
- 原因：`restart` 只是重启容器进程，配置和挂载已经在容器创建时确定了
- 解决：用 `docker compose up -d --force-recreate <服务名>` 重新创建容器
- 实例：改了 `.env.server` 里的 `ENVIRONMENT='dev'` 后，必须 force-recreate 才生效

**坑：`--force-recreate` 时端口映射变了**
- 原因：执行 `--force-recreate` 时如果没带 `--env-file`，会用默认值，导致端口映射变化
- 解决：始终带上 `--env-file deploy/backend/docker-compose/.env.docker`

#### 常用命令

| 命令 | 作用 | 是否危险 |
|------|------|---------|
| `docker ps` | 列出运行中的容器 | 只读 |
| `docker ps -a` | 列出所有容器（含已停止的） | 只读 |
| `docker compose ps` | Compose 管理的容器列表 | 只读 |
| `docker logs <容器名>` | 看容器日志（一次性） | 只读 |
| `docker logs -f <容器名>` | 实时跟踪日志（Ctrl+C 退出） | 只读 |
| `docker exec -it <容器名> bash` | 进入容器内部 | 只读（除非你在里面操作） |
| `docker stop <容器名>` | 停止容器 | 安全（数据保留） |
| `docker start <容器名>` | 启动已停止的容器 | 安全 |
| `docker rm <容器名>` | 删除容器 | **会删除容器（数据卷保留）** |

#### 实战示例

```bash
# 进入 PostgreSQL 容器看数据库
docker exec -it fba_postgres psql -U postgres -d fba

# 进入 Redis 容器
docker exec -it fba_redis redis-cli

# 看后端服务的日志
docker logs -f fba_server
```

### 2.3 Volume（数据卷）

#### 核心概念

**Volume = 容器的"外置硬盘"**。
- 类比：容器是租来的房子（拆了就没了），Volume 是你自己买的家具（放进房子，房子拆了家具还在）
- **关键问题**：容器删除后，里面的数据会丢；Volume 把数据存在容器外，让数据持久化

**三种数据持久化方式**：

| 方式 | 用法 | 适用场景 |
|------|------|---------|
| **Volume**（推荐） | `fba_postgres:/var/lib/postgresql/data` | 数据库数据、需要 Docker 管理 |
| **Bind Mount** | `./logs:/var/log/fba` | 配置文件、日志、源代码挂载 |
| **tmpfs** | 内存中的临时文件 | 敏感数据、临时缓存 |

**Day 1 实际用到的 Volumes**（来自 `docker-compose.yml`）：
- `fba_postgres` — PostgreSQL 数据（数据库表、记录都在这里）
- `fba_redis` — Redis 数据（虽然 Redis 主要是内存，但 RDB 持久化文件在这）
- `fba_rabbitmq` — RabbitMQ 消息和元数据
- `fba_static` — 静态资源（前端打包文件）
- `fba_static_upload` — 用户上传文件
- `fba_prometheus_data` — Prometheus 监控数据

**Bind Mount 实际示例**（`docker-compose.yml` 里 fba_server 的 volumes）：
```yaml
volumes:
  - ./deploy/backend/docker-compose/.env.server:/fba/backend/.env  # 配置文件
  - fba_static:/fba/backend/app/static                              # Volume
  - ${DOCKER_FBA_SERVER_LOG:-./logs/fba}:/var/log/fba              # 日志目录
```
- 第一行：把宿主机的 `.env.server` 文件挂进容器（改宿主机文件 → 容器里也变）
- 第二行：用名为 fba_static 的 Volume
- 第三行：把宿主机 `./logs/fba` 目录挂进容器，容器写日志 → 你能在宿主机看到

#### 踩坑记录

**坑：`docker compose down -v` 会删除 Volume**
- 原因：`-v` 标志表示"连同 volumes 一起删除"
- 后果：数据库表、Redis 缓存全部清空
- 解决：除非要重置环境，否则**不要加 `-v`**

**坑：改了挂载的 .env.server 文件后必须重建容器**
- 原因：bind mount 在容器创建时确定，运行中改源文件容器内是会同步，但应用程序通常只在启动时读一次配置
- 解决：`docker compose up -d --force-recreate fba_server`

#### 常用命令

| 命令 | 作用 | 是否危险 |
|------|------|---------|
| `docker volume ls` | 列出所有 volumes | 只读 |
| `docker volume inspect <名称>` | 看 volume 详情 | 只读 |
| `docker volume rm <名称>` | 删除 volume | **会永久删除数据** |
| `docker volume prune` | 删除所有未使用的 volumes | **危险，会删数据** |

### 2.4 网络（Network）

#### 核心概念

**Docker 网络 = 容器之间互相找到对方的方式**。
- 类比：每个容器是一个员工，Docker 网络是公司内网；同一个网络里的容器可以用"工号"（容器名）互相找到对方
- **关键事实**：默认情况下，容器之间是隔离的，必须放到同一个网络里才能互相通信

**三种主要网络模式**：

| 模式 | 用法 | 容器间通信 |
|------|------|---------|
| **bridge**（默认） | Docker 创建一个虚拟网桥 | 同网络下用容器名互相访问 |
| **host** | 直接用宿主机网络 | 没有隔离，性能最好但不安全 |
| **overlay** | 跨多台宿主机 | 多机集群（Docker Swarm） |

**Day 1 项目用的网络**（来自 `docker-compose.yml`）：
```yaml
networks:
  fba_network:
    name: fba_network
    driver: bridge
    ipam:
      config:
        - subnet: 172.10.10.0/24
```
- 创建了一个名为 `fba_network` 的 bridge 网络
- 所有 14 个服务都加入这个网络（每个服务的 `networks: - fba_network`）
- 容器之间用**容器名作为 hostname** 互相访问

**关键例子（连接字符串）**：
- 在 fba_server 容器里访问数据库：`postgresql://postgres:123456@fba_postgres:5432/fba`
  - 注意：是 `fba_postgres`，**不是 localhost**
  - 因为 fba_server 容器里的 localhost = 它自己，不是 PostgreSQL
- 在 fba_server 容器里访问 Redis：`redis://fba_redis:6379`
- 在 fba_server 容器里访问 RabbitMQ：`amqp://guest:guest@fba_rabbitmq:5672//`

**端口映射 vs 容器内端口（重要）**：
```yaml
ports:
  - "${DOCKER_MAP_POSTGRES_PORT:-5432}:5432"
```
- 冒号左边（5432）：**宿主机端口**，从你的电脑访问用这个
- 冒号右边（5432）：**容器内端口**，容器内部服务监听的端口
- Navicat 从你的电脑（宿主机）连数据库 → 用左边的 5432
- fba_server 容器内连数据库 → 用容器内端口 5432，但通过 `fba_postgres` 这个 hostname

#### 踩坑记录

**坑：在 Navicat 里用 `fba_postgres` 这个名字连不上**
- 原因：`fba_postgres` 只在 Docker 内部网络里能解析，宿主机不认识这个名字
- 解决：从宿主机连数据库要用 `localhost:5432`（端口映射后的端口）

**坑：端口映射没有自动加载导致连接失败**
- 原因：执行 `docker compose up --force-recreate` 时没指定 `--env-file`
- 解决：始终带 `--env-file deploy/backend/docker-compose/.env.docker`

#### 常用命令

| 命令 | 作用 |
|------|------|
| `docker network ls` | 列出所有网络 |
| `docker network inspect fba_network` | 看网络详情（哪些容器在里面） |

### 2.5 多阶段构建（Multi-stage Build）

#### 核心概念

**多阶段构建 = 用一个 Dockerfile 分多个阶段构建，最终镜像只保留需要的东西**。
- 类比：装修房子时先用脚手架（builder）盖好，然后拆掉脚手架（最终镜像没有脚手架）
- 解决的痛点：镜像里有大量"构建时需要、运行时不需要"的东西（编译器、源码、缓存），导致镜像巨大

**Day 1 项目的 Dockerfile 结构**：
```
阶段 1：builder（uv:python3.10-bookworm-slim）
  - 安装编译工具（gcc）
  - 用 uv 安装所有 Python 依赖
  - 这个阶段镜像很大（几个 G）

阶段 2：base_server（python:3.10-slim-bookworm）
  - 只装运行时需要的（curl、supervisor）
  - 从阶段 1 复制 /usr/local/bin/uv
  - 从阶段 1 复制 /usr/local（已经装好的 Python 包）
  - 复制 supervisor 配置

阶段 3-6：fba_server / fba_celery_worker / fba_celery_beat / fba_celery_flower
  - 都从 base_server 继承
  - 各自加上对应的 supervisor 配置和暴露端口
```

**最终镜像里有什么**：
- 阶段 1 的内容**完全丢掉**（gcc、构建缓存都没了）
- 只保留从阶段 1 `COPY --from=builder` 复制进来的东西
- 镜像体积大幅缩小

**为什么这么做**：
1. **减小镜像体积**：阶段 1 几个 G，最终阶段可能只有 1G
2. **加快传输**：镜像小，pull/push 快
3. **减少攻击面**：运行时不带编译器，被攻破时风险小

#### 代码片段

```dockerfile
# === 阶段 1：构建（带 uv）===
FROM ghcr.io/astral-sh/uv:python3.10-bookworm-slim AS builder

RUN apt-get update && apt-get install -y gcc python3-dev

COPY . /fba
WORKDIR /fba

# 用 uv 装依赖（带 BuildKit cache 加速）
RUN --mount=type=cache,target=/root/.cache/uv \
    uv sync --locked --no-default-groups --group server --no-install-project

# === 阶段 2：运行时基础 ===
FROM python:3.10-slim-bookworm AS base_server

RUN apt-get update && apt-get install -y curl supervisor

# 关键：从 builder 阶段复制
COPY --from=builder /usr/local/bin/uv /usr/local/bin/uv
COPY --from=builder /fba /fba
COPY --from=builder /usr/local /usr/local

# === 阶段 3：FastAPI 服务 ===
FROM base_server AS fba_server
COPY deploy/backend/supervisor/fba_server.conf /etc/supervisor/conf.d/
EXPOSE 8001
CMD ["supervisord", "-c", "/etc/supervisor/supervisord.conf"]
```

**关键关键字**：
- `FROM ... AS <名称>` — 给阶段起名字
- `COPY --from=<阶段名>` — 从指定阶段复制内容
- `--mount=type=cache` — BuildKit 缓存挂载（加速依赖安装）

## 三、Docker Compose

### 3.1 服务编排

#### 核心概念

**Docker Compose = 用一个 YAML 文件定义和管理多个容器**。
- 类比：单独 `docker run` 像一个个手动启动 14 台机器；`docker compose up` 像按下"全场启动"按钮
- 一个 `docker-compose.yml` 文件描述：
  - **services**（服务）— 要起哪些容器
  - **networks**（网络）— 容器之间怎么连
  - **volumes**（卷）— 数据怎么持久化

**核心结构**：
```yaml
services:        # 定义所有容器
  fba_server:
    build: ...   # 怎么构建
    image: ...   # 用哪个镜像
    ports: ...   # 暴露什么端口
    volumes: ... # 挂载什么
    networks: ... # 加入哪个网络
    depends_on: ... # 依赖哪些服务
    environment: ... # 环境变量

networks:        # 定义网络
  fba_network:
    driver: bridge

volumes:         # 定义命名卷
  fba_postgres:
```

**Day 1 跑过的核心命令**：

| 命令 | 作用 | 是否危险 |
|------|------|---------|
| `docker compose up -d` | 后台启动所有服务 | 安全 |
| `docker compose up -d <服务名>` | 只启动某个服务 | 安全 |
| `docker compose ps` | 看所有服务状态 | 只读 |
| `docker compose logs -f <服务>` | 实时看日志 | 只读 |
| `docker compose stop` | 停止所有容器（保留） | 安全 |
| `docker compose start` | 重新启动已停止的容器 | 安全 |
| `docker compose restart` | 重启容器（不重新加载挂载） | 安全 |
| `docker compose down` | 停止并删除容器（保留 volumes） | 一般安全 |
| `docker compose down -v` | 停止并删除容器**和数据** | **危险！会删数据** |
| `docker compose up -d --force-recreate` | 强制重建容器 | 安全（数据保留） |
| `docker compose up -d --build` | 重新构建镜像并启动 | 安全 |

#### 配置示例

**完整一个服务的写法**（来自本项目 fba_server）：
```yaml
fba_server:
  build:
    context: .
    dockerfile: Dockerfile
  image: fba_server:latest
  ports:
    - "${DOCKER_MAP_SERVER_PORT:-8001}:8001"
  container_name: fba_server
  restart: always
  environment:
    - TZ=Asia/Shanghai
  depends_on:
    - fba_postgres
    - fba_redis
  volumes:
    - ./deploy/backend/docker-compose/.env.server:/fba/backend/.env
    - fba_static:/fba/backend/app/static
    - ${DOCKER_FBA_SERVER_LOG:-./logs/fba}:/var/log/fba
  networks:
    - fba_network
  command:
    - bash
    - -c
    - |
      wait-for-it -s fba_postgres:5432 -s fba_redis:6379 -t 300
      supervisord -c /etc/supervisor/supervisord.conf
```

**逐项解释**：
- `build` — 用当前目录的 Dockerfile 构建（context 是构建上下文）
- `image` — 构建后的镜像叫什么
- `ports: "8001:8001"` — 宿主机 8001 → 容器 8001
- `${DOCKER_MAP_SERVER_PORT:-8001}` — 从 .env 文件读端口，没设就用 8001
- `container_name` — 给容器起个固定名字（默认会随机生成）
- `restart: always` — 容器挂了自动重启
- `depends_on` — 启动顺序：先启动 postgres 和 redis，再启动 server（**但不保证服务真的可用**）
- `volumes` — 挂载配置文件、静态资源、日志目录
- `command` — 容器启动后执行的命令；这里先 wait-for-it 等数据库就绪，再启动 supervisord

---

### 3.2 depends_on

#### 核心概念

**depends_on 只保证启动顺序，不保证服务可用！**

- 类比：`depends_on: fba_postgres` = "我要等 postgres **进程启动**了再启动"，但 postgres 进程启动 ≠ 数据库准备好接受连接
- 数据库进程启动 → 加载配置 → 初始化数据目录 → 监听端口 → **真正可用**（这中间可能 10-30 秒）
- 如果 fba_server 在 postgres 还没准备好时就启动，会立刻连数据库失败

**正确的解决方案：wait-for-it / healthcheck**

本项目用 `wait-for-it`：
```yaml
command:
  - bash
  - -c
  - |
    wait-for-it -s fba_postgres:5432 -s fba_redis:6379 -t 300
    supervisord -c /etc/supervisor/supervisord.conf
```
- `wait-for-it -s fba_postgres:5432` — 持续探测 5432 端口，直到能建立 TCP 连接
- `-t 300` — 最多等 300 秒，超时就放弃
- 探测成功后才执行下一行 `supervisord ...`

**更严格的方案**（本项目没用，但了解一下）：在被依赖的服务上配 `healthcheck`：
```yaml
fba_postgres:
  healthcheck:
    test: ["CMD", "pg_isready", "-U", "postgres"]
    interval: 5s
    timeout: 3s
    retries: 5

fba_server:
  depends_on:
    fba_postgres:
      condition: service_healthy   # 等 postgres 健康检查通过才启动
```

#### 踩坑记录

**坑：服务启动顺序对了，但应用还是连不上数据库**
- 原因：以为 `depends_on` 万能，没意识到它只管"进程启动"
- 解决：用 `wait-for-it` 或 `healthcheck` 等到服务真正可用

### 3.3 网络配置

#### 核心概念

**Compose 自动给所有服务加入同一个默认网络**，所以即使不写 `networks:` 也能互通。但本项目显式定义了网络以便控制：

```yaml
networks:
  fba_network:
    name: fba_network        # 网络名（不指定的话会加 compose 项目前缀）
    driver: bridge           # 桥接模式（默认）
    ipam:
      driver: default
      config:
        - subnet: 172.10.10.0/24   # 自定义子网
```

**关键规则**：
1. 同一网络内的容器，可以用**容器名**或**服务名**作为 hostname 互相访问
2. 不在同一网络的容器无法直接通信
3. 宿主机访问容器要靠 `ports` 端口映射

**Day 1 跑通的实际访问路径**：
```
你的浏览器
  → http://localhost:8000        （宿主机 8000 端口）
  → fba_nginx 容器（80 端口）    （nginx 容器内）
  → http://fba_server:8001       （Docker 内部网络访问）
  → fba_server 容器（FastAPI）
  → fba_postgres:5432            （Docker 内部网络访问数据库）
```

#### 踩坑记录

**坑：宿主机连容器要用 localhost，容器之间互连用容器名**
- 错误：在 Navicat 里写 hostname 为 `fba_postgres`
- 正确：在 Navicat 里写 `localhost` 或 `127.0.0.1`
- 原因：Navicat 跑在你的宿主机上，不在 Docker 网络里

### 3.4 环境变量文件（.env）

#### 核心概念

**Docker Compose 项目里有两类 .env 文件，作用完全不同**：

| 文件 | 位置 | 谁读取 | 作用 |
|------|------|--------|------|
| `.env.docker` | `deploy/backend/docker-compose/.env.docker` | docker compose 命令本身 | 控制端口映射、卷路径等 Compose 配置 |
| `.env.server` | `deploy/backend/docker-compose/.env.server` | FastAPI 应用代码 | 控制应用行为（开发/生产、数据库密码等） |

**关键理解**：

**`.env.docker` 用法**（被 Compose 读取，影响 YAML 文件）：
```yaml
ports:
  - "${DOCKER_MAP_POSTGRES_PORT:-5432}:5432"
```
- `${...}` 是 Compose 在解析 YAML 时替换的，从 .env.docker 读
- `:-5432` 是默认值（变量没设就用这个）
- 必须用 `--env-file deploy/backend/docker-compose/.env.docker` 指定

**`.env.server` 用法**（被挂载进容器，给 FastAPI 应用读）：
```yaml
volumes:
  - ./deploy/backend/docker-compose/.env.server:/fba/backend/.env
```
- 这是把宿主机的文件挂进容器
- 容器内的 FastAPI 用 `pydantic-settings` 读取 `/fba/backend/.env`
- 改这个文件控制：`ENVIRONMENT='dev'/'prod'`、数据库连接、Redis 配置等

**两种"环境变量"的区别**：
1. **Compose 级别**（.env.docker）：只影响 docker-compose.yml 怎么解析
2. **应用级别**（.env.server）：只影响应用代码读到什么配置

#### 踩坑记录

**坑：改了 `.env.server` 里的 `ENVIRONMENT='prod'` 为 `'dev'` 但 Swagger 还是 400**
- 原因：`docker compose restart` 不会重新挂载，应用程序也没重启读配置
- 解决：`docker compose up -d --force-recreate fba_server`

**坑：`--force-recreate` 没带 `--env-file`，端口映射变了**
- 原因：Compose 找不到 `.env.docker`，所有 `${...}` 用了默认值
- 实例：本来端口映射是 15432，结果变成默认的 5432
- 解决：完整命令 `docker compose --env-file deploy/backend/docker-compose/.env.docker up -d --force-recreate fba_server`

### 3.4 Docker 端口映射的本质（重要）

#### 核心概念

**端口映射 ≠ 把容器 Redis 变成本地 Redis**

Docker 的 `6379:6379` 只是说：

> 把容器里的 6379，**暴露**给宿主机的 6379

容器里的 Redis 还是容器里的 Redis，只是宿主机也能访问它了。

**你电脑里实际同时存在两个 Redis：**

```
┌──────────────────────────────────────┐
│  Windows 宿主机                       │
│                                      │
│  Redis A（本地 Windows Redis）        │
│  进程：redis-server.exe               │
│  监听：127.0.0.1:6379                 │
│  密码：cjk123                         │
│                                      │
│  ┌────────────────────────────────┐  │
│  │  Docker Desktop                │  │
│  │                                │  │
│  │  Redis B（fba_redis 容器）      │  │
│  │  监听：容器内 0.0.0.0:6379      │  │
│  │  端口映射 → 宿主机 0.0.0.0:6379 │  │
│  │  密码：无                       │  │
│  │                                │  │
│  └────────────────────────────────┘  │
└──────────────────────────────────────┘
```

#### localhost 和 127.0.0.1 为什么不一样？

`localhost` 是"域名"，`127.0.0.1` 是"IP"，Windows 解析时有优先级差异：

```
host="localhost"
    ↓
Windows 优先解析成 ::1（IPv6 回环）
    ↓
命中 Docker Redis 的 [::]:6379
    ↓
Docker Redis（无密码）
    ↓
你传了 password="cjk123" → 报错：AUTH called without any password

host="127.0.0.1"
    ↓
直接走 IPv4
    ↓
命中 Windows Redis 的 127.0.0.1:6379
    ↓
Windows Redis（有密码 cjk123）
    ↓
连接成功
```

**用 `netstat` 看清楚谁在监听什么：**

```
TCP    0.0.0.0:6379    LISTENING  27336  ← Docker Redis（抢了所有 IPv4）
TCP    127.0.0.1:6379  LISTENING  7124   ← Windows Redis（只占 IPv4 回环）
TCP    [::]:6379       LISTENING  27336  ← Docker Redis（IPv6，localhost 会走这里）
```

#### 结论

| 写法 | 实际连到 | 密码 |
|------|---------|------|
| `host="localhost"` | Docker Redis（IPv6） | 无密码 |
| `host="127.0.0.1"` | Windows Redis（IPv4） | cjk123 |

### 9.4 网络

#### 核心概念

**curl 是什么**：命令行 HTTP 客户端，用来手动发各种 HTTP 请求，测试接口必备工具。

**基本结构**：
```
curl [方法] [地址] [请求头] [请求体]
```

**常用参数**：

| 参数 | 含义 | 示例 |
|------|------|------|
| `-X GET/POST/DELETE` | 指定 HTTP 方法，不写默认 GET | `-X POST` |
| `-H "key: value"` | 添加请求头，可多个 | `-H "Content-Type: application/json"` |
| `-d "{...}"` | 请求体，只用于 POST/PUT | `-d "{\"title\": \"test\"}"` |

**实际例子**：

```bash
# POST 创建笔记（有请求体）
curl -X POST http://127.0.0.1:8002/notes -H "Content-Type: application/json" -d "{\"title\": \"test\", \"content\": \"hello\"}"

# GET 获取笔记（带认证 token）
curl -X GET http://127.0.0.1:8002/notes -H "authorization: Bearer eyJhbG..."

# DELETE 退出登录（带认证 token）
curl -X DELETE http://127.0.0.1:8002/auth/logout -H "authorization: Bearer eyJhbG..."

# POST 登录（有请求体，不需要 token）
curl -X POST http://127.0.0.1:8002/auth/login -H "Content-Type: application/json" -d "{\"username\": \"luchia\", \"password\": \"123456\"}"
```

**规律**：
- 需要认证的接口 → 加 `-H "authorization: Bearer 你的token"`
- 发 JSON 数据 → 加 `-H "Content-Type: application/json"` + `-d "{...}"`
- GET/DELETE → 通常不需要 `-d`

#### 踩坑记录

**踩坑1：localhost 和 127.0.0.1 在 Windows + Docker 环境下不等价**

- 现象：`redis.Redis(host="localhost")` 连接失败，改成 `host="127.0.0.1"` 才正常
- 原因：Docker 占了 `0.0.0.0:6379`（监听所有网卡），Windows 解析 `localhost` 时走 IPv6（`::1`），匹配到 Docker 的 `[::]:6379`（无密码）；而本地 Windows Redis 只监听 `127.0.0.1:6379`（有密码），必须用 IPv4 地址才能连到它
- 排查方法：`netstat -ano | findstr :6379` 看哪个 PID 在监听哪个地址
- 结论：有 Docker 的 Windows 机器上，连本地 Redis 要用 `127.0.0.1` 而不是 `localhost`

**踩坑2：bare `except:` 吞掉 Redis 写入失败**

- 现象：登录返回 200 和 token，但 Redis 里没有 key
- 原因：`r.set()` 抛出 `AuthenticationError`，被 `except:` 静默吞掉，函数继续执行返回 token
- 修复：`except:` 改为 `except HTTPException: raise`，其他异常单独处理，不要用裸 except 吞掉所有错误

- 预防：重建容器时始终带上 `--env-file deploy/backend/docker-compose/.env.docker`

**坑4：登录报 404 用户名或密码有误**
- 原因：数据库表结构是空的，没有执行初始化数据脚本
- 解决：执行 `backend/sql/postgresql/init_test_data.sql` 写入默认数据
- 默认账号：username=`admin`，password=`123456`

   - 提出假设
   - 设计验证方案
   - 执行验证

5. **修复与验证**
   - 实施修复
